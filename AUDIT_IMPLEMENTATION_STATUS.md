# Proofound Implementation Audit Against Canonical MVP Docs

## 1. Executive Summary

- Overall status: substantial MVP implementation exists, but the repository is not launch-capable against the current canonical docs.
- Rough completion for in-scope MVP: about 63%.
- Launch view: not launch-capable.
- Biggest risks:
  - Canonical org role and membership model is not implemented as specified. Code still centers `owner/admin/member/viewer` instead of the canonical `org_owner/org_manager/org_reviewer` model and related membership-state rules.
  - Upload security and file-contract behavior contradict the canonical upload contract on file types and per-file limits.
  - Privacy and authorization are mixed between newer server-side contracts and older broad RLS and privacy helper patterns, so deny-by-default is not yet provable end to end.
  - Launch operations evidence is incomplete. Monitoring, smoke coverage, rollback support, and restore-drill proof are only partially wired.
  - Some surfaces look complete in UI or route shape but still rely on TODO logic, placeholders, compatibility fallbacks, or weak tests.

What is clearly implemented and working:

- Individual and org onboarding shells, dashboard surfaces, expertise atlas, verification status, public portfolio publication, user export/delete flows, feature-flag APIs, and launch/perf scripts all have real code paths.
- The codebase contains substantial canonical schema support in `src/db/schema.ts` for proof packs, artifacts, verification, submissions, lifecycle operations, exports, deletions, reveal events, feature flags, Zen partitioning, and launch metrics.
- Matching review and reveal logic is materially implemented, not just mocked, especially in `src/lib/matching/review-contract.ts`.

What keeps this below launch:

- Canonical permission model drift, upload-policy drift, incomplete operational hardening, and several partial workflow paths around interview/feedback/review and privacy enforcement.

## 5. Contradictions

1. Org role model contradicts the canonical role matrix.
   - Canonical docs: `PRD_for_a_web_platform_MVP.master-latest.md` Sections `2.6` and `4.1B` define `org_owner`, `org_manager`, and `org_reviewer`.
   - Code: `src/lib/authz/policy.ts`, `src/actions/org.ts`, `src/lib/auth/admin.ts`, and `src/db/policies.sql` use `owner/admin/member/viewer`.
   - Impact: permissions, invites, ownership transfer, reviewer scope, audit semantics, and launch smoke expectations are all harder to prove against the canonical contract.

2. Upload contract contradicts canonical file-type and limit rules.
   - Canonical docs: master PRD `4.1D` allows PDF, PNG, JPEG, WebP, text/plain, markdown, and `25 MB` per file.
   - Code: `src/lib/uploads/lifecycle.ts` caps files at `10 MB` by kind, allows `application/msword` and HEIF/HEIC, and does not admit text/plain or markdown.
   - Impact: real user behavior will not match the documented launch contract.

3. Interview provider contract contradicts the master PRD.
   - Canonical docs: master PRD `4.2.D` says interview scheduling supports Google Meet and manual links; Zoom may appear only as fallback guidance.
   - Code: `src/app/api/interviews/schedule/route.ts` is documented and wired as live Zoom-or-Google scheduling support.
   - Impact: scope and fallback posture are misaligned.

4. Legacy RLS posture conflicts with deny-by-default privacy intent.
   - Canonical docs: master PRD `4.1A`, `4.1B`; tech NFR `1.1`, `1.2`.
   - Code: `src/db/policies.sql` still includes broad policies such as `CREATE POLICY "Users can view all profiles" ... USING (true);`.
   - Impact: even where higher-layer visibility logic exists, the lowest-level policy posture does not read as canonical deny-by-default.

5. Smoke test expectations contradict the actual runtime and runbook.
   - Canonical docs: runbook Section `2` requires launch smoke checks for critical paths.
   - Code: `tests/e2e/smoke.spec.ts` expects `/api/health` to return `404`, while the repo contains `src/app/api/health/route.ts`.
   - Impact: smoke coverage is not trustworthy as launch evidence.
