# Project Change Entry

- Date/time (UTC): 2026-03-09T10:44:02Z
- Branch: master
- Base commit: a8d1c42b

What changed:

- Changed `next.config.js` so Sentry no longer widens client sourcemap upload by default during Vercel builds.
- Added an explicit opt-in env switch, `SENTRY_WIDEN_CLIENT_FILE_UPLOAD=1`, for cases where extra client stacktrace context is worth the deploy-time cost.
- Documented the new behavior in `docs/sentry-setup.md`.

Why:

- Local Vercel-mode builds complete in about 85 to 97 seconds, so the reported 30+ minute production deploy was not caused by `next build` itself.
- The repo currently emits hundreds of sourcemap files, and widened Sentry client uploads were the most likely deploy-only overhead left in the Vercel build path.

How to verify:

- `npm run lint`
- `npm run typecheck`
- `npm run docs:freshness`
- `env VERCEL=1 VERCEL_ENV=production NEXT_TELEMETRY_DISABLED=1 npm run build`
- On the next production deploy, confirm Vercel build time drops materially. If deeper client sourcemap coverage is needed later, set `SENTRY_WIDEN_CLIENT_FILE_UPLOAD=1` and compare deploy duration.

Open risks / TODO:

- This keeps standard Sentry sourcemap upload enabled, but removes the widened client upload surface by default, so some minified client frames may have less context than before.
- Vercel build logs were not directly inspectable from this shell because the Vercel CLI is not authenticated.
