# Project Change Entry

- Date/time (UTC): 2026-03-11T07:19:44Z
- Branch: master
- Base commit: d254db38

What changed:

- Added a Vercel-only webpack cache guard in `next.config.js` so Vercel builds do not restore or write webpack filesystem pack caches.
- Disabled Sentry build-time sourcemap processing for Vercel production builds unless explicitly re-enabled with `SENTRY_ENABLE_BUILD_SOURCEMAPS=1`.
- Kept the existing stale-deployment pruning and duplicate retry protections unchanged.

Why:

- Recent production deploys were failing or stalling after restoring oversized webpack pack caches on Vercel.
- A forced no-cache deploy avoided cache reuse but still ran too long and eventually failed, so the build path itself needed to stop using filesystem cache on Vercel.
- Sentry sourcemap work is non-essential during this stabilization phase and adds build-time overhead.

How to verify:

- Run `VERCEL=1 VERCEL_ENV=production npm run build` and confirm the build completes.
- Deploy to Vercel production and confirm the build log no longer shows cache cleanup and does not fail with webpack pack-file OOM symptoms.
- Inspect the latest production deployment and confirm only one active production deployment is running for `master`.

Open risks / TODO:

- This is the stabilization phase only. The planned multi-zone split into marketing, product, and API apps is still pending.
- If Vercel production still OOMs after this change, the next step is to split the API surface out before any further forced cold deploys.
