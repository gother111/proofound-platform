# Project Change Entry

- Branch: master

What changed:

- Hardened auth signup behavior in `src/actions/auth.ts` to preserve email-verification semantics under provider failures.
- Removed admin auto-confirm fallback for signup throttle/email-send failures.
- Added retry-safe handling for Supabase SMTP confirmation errors (`Error sending confirmation email`) and rate-limit conditions.
- Extended email verification action to support OTP types from callback context (`email` and `signup`).
- Updated callback routing in `src/app/auth/callback/route.ts`:
  - verification redirects now support both `type=email` and `type=signup`
  - recovery redirect no longer forwards `code` to reset-confirm URL
- Updated reset confirm bootstrap in `src/app/(auth)/reset-password/confirm/ConfirmResetPasswordForm.tsx` to avoid duplicate recovery code exchange and to support token-hash recovery bootstrap.
- Updated verify-email client flow in `src/app/(auth)/verify-email/VerifyEmailContent.tsx` to pass OTP type through to server action.
- Added callback route regression test file `tests/api/auth-callback-route.test.ts`.
- Expanded `tests/actions/auth.test.ts` coverage for:
  - signup throttle semantics (no auto-confirm fallback)
  - Supabase SMTP confirmation-send fallback via admin-generated link + Resend API
  - Supabase SMTP recovery-send fallback via admin-generated link + Resend API
  - verifyEmail OTP type handling (`email`, `signup`)
  - explicit reset redirect URL assertion
- Aligned mock Supabase auth surfaces for browser/server mocks by adding `exchangeCodeForSession`, `verifyOtp`, `resetPasswordForEmail`, and `resend` methods where missing.
- Updated docs:
  - `agent/runbooks/setup.md` with canonical auth-email transport rule and fallback behavior
  - `agent/checklists/verification.md` with manual SMTP auth-email smoke checklist and fallback checks.

Why:

- PRO-61 required validating and fixing signup/reset auth email flows after new Supabase SMTP integration to Resend.
- Existing flow risked bypassing verification through admin auto-confirm fallback.
- Recovery callback/confirm flow had code-exchange sequencing that could invalidate recovery links.
- Callback verification routing did not explicitly support both OTP verification types used by Supabase links.
- Supabase SMTP currently returns provider failures for real auth sends (`Error sending confirmation email` / `Error sending recovery email`), so fallback delivery path was required to keep auth email flows operational.
