# Project Change Entry

- Date/time (UTC): 2026-03-11T06:37:33Z
- Branch: master
- Base commit: a8bf6931
  What changed:
- Stopped automatic `.next/cache` deletion during every Vercel build. Cache cleanup is now opt-in via `NEXT_CLEAN_BUILD_CACHE=1`.
- Added `scripts/cancel-stale-vercel-deployments.mjs` plus `npm run vercel:cancel-stale` to detect and delete superseded queued or building Vercel deployments per branch and target.
- Added `.github/workflows/cancel-stale-vercel-deployments.yml` so pushes and PR updates prune stale Vercel deploys automatically.
- Hardened `.github/workflows/retry-vercel-deploy.yml` to cancel stale production deploys first, avoid duplicate retries for a SHA that is already building, and cancel older in-progress retry runs.
- Updated `agent/checklists/verification.md` with the new deployment-pruning command.

Why:

- Vercel showed a deployment storm with one active production build and 21 stale queued production deploys, while older production deploys had already failed after 29 to 46 minutes.
- Forced cache eviction on every Vercel build removed Next.js incremental build reuse, which likely caused the jump from normal 4 to 6 minute deploys to much longer failing deploys.
- The retry workflow could also add fresh production deploys while an identical SHA was already queued or building.

How to verify:

- `npm run vercel:cancel-stale -- --target production --branch master --limit 10`
- `npm run docs:freshness`
- `npm run lint`
- `npm run typecheck`
- `VERCEL=1 npm run build`
- `npx -y vercel@latest ls proofound-platform --token "$VERCEL_TOKEN"`

Open risks / TODO:

- Vercel preview deployments for many distinct branches can still queue if the account concurrency limit is saturated. This change removes stale duplicates and restores caching, but it cannot remove account-level concurrency limits.
- Existing queued GitHub Actions runs for `Retry Vercel Deploy Until Synced` could not be canceled via the GitHub API and returned HTTP 500.
