# Project Change Entry

- Date/time (UTC): 2026-03-09T19:48:54Z
- Branch: master
- Base commit: d700ee77
  What changed:
- Tightened the individual readiness engine to compute the new trust ladder: `discoverable`, `match_visible`, `intro_eligible`, and `strongly_trusted`, while keeping legacy readiness states for compatibility.
- Updated matching eligibility and intro-interest APIs to return structured trust and intro-eligibility payloads, including `409 INTRO_QUALIFICATION_NOT_MET` for blocked intro attempts.
- Refreshed dashboard trust copy and the core PRD, technical requirements, and launch-operations docs to replace Lite/Strong activation language with the stronger qualified-introduction policy.

Why:

- The old intro gate was too weak for a proof-first product. Browsing and match exploration needed to stay low-friction while credible introductions required deeper, fresher, and at least partly trusted proof.

How to verify:

- `npm run typecheck`
- `npm run test -- src/lib/__tests__/matching-eligibility.test.ts tests/api/match-interest-route.test.ts tests/api/core-matching-gating-routes.test.ts tests/actions/onboarding.test.ts`
- `npm run lint`
- `npm run docs:freshness`

Open risks / TODO:

- Assignment-level relevance still uses assignment `mustHaveSkills` overlap as the MVP gate. A richer role-to-skill ontology is still a future improvement.
- Shortlist review payloads do not yet expose per-candidate trust-level fields; the main intro and browse surfaces now do.
