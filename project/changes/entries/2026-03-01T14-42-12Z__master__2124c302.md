What changed:

- Updated custom verification skill filtering in `GET /api/expertise/verifications/custom/artifacts` to treat skills as already verified only when accepted requests are integrity clear (`integrity_status` clear, null treated as clear).
- Updated selected-skill validation in `POST /api/expertise/verifications/custom/request` to use the same accepted + integrity-clear rule.
- Added four regression tests in `tests/api/custom-verification-routes.test.ts` for flagged-versus-clear accepted behavior in both GET artifacts and POST custom request paths.

Why:

- Users reported unverified skills not appearing in the Custom verification request dialog despite being unverified in the atlas.
- The custom flow used broader accepted-request filtering than the existing integrity-aware verification semantics elsewhere in the expertise stack.
