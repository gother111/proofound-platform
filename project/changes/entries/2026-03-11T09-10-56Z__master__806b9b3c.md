# Project Change Entry

- Date/time (UTC): 2026-03-11T09:10:56Z
- Branch: master
- Base commit: 806b9b3c
  What changed:
- Removed the unused multi-project API split artifacts, including `apps/api/**`, the Vercel API-project bootstrap scripts, and the API-project runbook.
- Added `scripts/vercel-should-build.mjs` and wired it into `vercel.json` as the single-project ignored build step so docs, governance, and other non-runtime-only changes can skip Vercel builds safely.
- Added the manual `Prepare Release Candidate` workflow plus the `Enforce Release Branches Into Master` PR gate so production batches can be assembled on `release/*` branches and merged to `master` deliberately.
- Updated `package.json`, `agent/checklists/verification.md`, and added `agent/runbooks/release-batch-flow.md` to document and verify the new single-project release process.
- Deleted the live `proofound-platform-api` Vercel project so `proofound-platform` is again the only Vercel project in use.

Why:

- The user wanted one Vercel project, smaller production batches, and fewer unnecessary deployments, not a multi-project routing split.
- Release batching through `release/*` branches reduces deploy queue pressure without changing public routing, cookies, or API contracts.
- The ignored build step prevents docs-only, audit-only, and agent-log-only changes from consuming Vercel build capacity.

How to verify:

- `node ./scripts/vercel-should-build.mjs --changed-files project/example.md` should exit `0`.
- `node ./scripts/vercel-should-build.mjs --changed-files src/app/page.tsx` should exit `1`.
- `node ./scripts/vercel-should-build.mjs --changed-files next.config.js` should exit `1`.
- `npm run lint`
- `npm run typecheck`
- `VERCEL=1 npm run build`
- `npm run docs:freshness`
- `gh workflow run "Prepare Release Candidate" -f release_name=<name> -f commit_shas=<sha1,sha2>`
