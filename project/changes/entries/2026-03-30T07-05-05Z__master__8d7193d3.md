# Project Change Entry

- Date/time (UTC): 2026-03-30T07:05:05Z
- Branch: master
- Base commit: 8d7193d3
  What changed:
- Switched the default production deployment path to GitHub Actions prebuilt deploys using `vercel pull`, `vercel build`, and `vercel deploy --prebuilt`.
- Added repo-native Vercel scripts in `package.json` and a shared helper in `scripts/vercel-command.mjs` for pull, build, and prebuilt deploy commands with Git metadata attached.
- Updated `scripts/cancel-stale-vercel-deployments.mjs` and `.github/workflows/retry-vercel-deploy.yml` so production deploy detection and cleanup work for CLI prebuilt deployments instead of assuming Git-sourced cloud builds.
- Updated `README.md`, `docs/deployment-guide.md`, `agent/runbooks/setup.md`, `agent/runbooks/release-batch-flow.md`, `agent/checklists/preflight.md`, and `agent/checklists/verification.md` so production-first prebuilt deployment is the documented default while preview Git deploys remain unchanged.
- Hardened the Vercel helper so it does not pass tokens as CLI arguments and redacts any token value that might appear in captured command output.

Why:

- The repo already treated `vercel pull` and `vercel build --prod` as the closest local parity checks, but production automation and top-level docs still depended on Vercel cloud builds.
- Making CI and local builds the source of truth reduces deployment drift and makes production rollout behavior easier to verify before upload.
- Keeping preview Git integration in place is the lowest-risk cutover because preview branch/domain behavior and OAuth expectations stay unchanged for now.

How to verify:

- `npm run lint`
- `npm run typecheck`
- `npm run docs:freshness`
- `npm run vercel:preflight`
- `npm run vercel:pull:production`
- `npm run vercel:build:production`
- Confirm `.vercel/output/config.json` and `.vercel/output/builds.json` exist.
- After merge to `master`, confirm the `Retry Vercel Deploy Until Synced` workflow runs and records a prebuilt deployment URL in the job summary.

Open risks / TODO:

- If Vercel Git auto-deploys remain enabled for production, they can still create duplicate cloud-build deployments alongside the new prebuilt workflow until the setting is intentionally changed in Vercel.
- Preview deployments remain Git-based. If the team later moves previews to prebuilt deploys, it should validate branch-specific domains, env handling, and OAuth callback assumptions first.
